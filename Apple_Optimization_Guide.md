# Apple Optimization Guide

## Overview
The `APPLE-002` feature introduces a Compiler-Native Device Optimization pass specifically targeting Apple Silicon deployments. This operates firmly within the `OptimizationPass` compiler framework, strictly transforming immutable `PlanningBundle` instances to produce `AppleOptimizationReport` artifacts.

## Architectural Boundaries
1. **Device Optimization vs. Execution Engine:** The Execution Engine never interprets Apple optimization policy. It continues to rely strictly on immutable `PhysicalIR` structures. Device placement mapping generated by Device Planning is optimized upstream inside the Compiler Pipeline via `AppleDeviceOptimizationPass`.
2. **Immutable Objects:** Policies (`AppleOptimizationPolicy`), artifacts (`PlacementOptimization`, `AffinityOptimization`), and the reports themselves (`AppleOptimizationReport`) are all strictly read-only after instantiation.

## Usage
When processing an Apple deployment:
1. `DevicePlanner` generates a `DevicePlan` outlining base capability and placement (`unified_memory`).
2. `CompilerPlanner` aggregates this into a `PlanningBundle`.
3. `AppleDeviceOptimizationPass` consumes the `PlanningBundle`, inspecting the attached `DevicePlan`.
4. It performs placement modifications and outputs a generic optimization summary (`OptimizationDiagnostics`, `OptimizationStatistics`) stored inside the `PlanningBundle` metadata structure.
5. `CompilerEngine` converts the `PlanningBundle` into its final form for backend processing, unhindered by Apple-specific logic.

## Future Extensibility
Because Apple Optimization integrates transparently with the Optimization Pipeline, future hardware accelerators (e.g., CUDA, ROCm) require only separate implementations of `OptimizationPass` referencing discrete policies. No fundamental Runtime or ExecutionEngine changes will be required.
