# Backend Manual

## 1. Overview
The execution backend is the final stage of the oMLX execution pipeline. It takes the backend-agnostic Physical IR generated by the lowering passes and translates it into concrete, executable operations (e.g., MLX, Metal, CUDA).

## 2. BackendDescriptor
The `BackendDescriptor` is an immutable specification of a backend's capabilities, topology, and execution constraints. It provides the necessary metadata for the `ExecutionPlanner` and `LoweringEngine` to construct a compatible graph.

### Core Fields
- **Identity:** `backend_id`, `backend_version`, `backend_family`, `backend_generation`.
- **Supported Paradigms:** `supported_execution_semantics`, `supported_operation_mappings`, `supported_execution_families`.
- **Cache & Synchronization:** `supported_cache_layouts`, `supported_synchronization_primitives`, `supported_cache_strategies`.
- **Execution & Routing:** `supported_execution_modes`, `supported_routing_strategies`, `supported_graph_features`.
- **Data & Precision:** `supported_quantization_formats`, `supported_precision_formats`.
- **Hardware & Memory:** `hardware_capabilities`, `hardware_metadata`, `memory_model`, `memory_topology`.
- **Topology:** `execution_topology`, `stream_model`, `device_topology`.
- **Extensibility:** `backend_metadata`.

## 3. Backend Capability
Backend capabilities represent the actual, measurable features supported by the backend instance running on the specific host hardware. They are used to filter and select backends during the planning phase.

## 4. AdapterRegistry
The `AdapterRegistry` holds `ModelAdapter`s, which encapsulate model-specific behavior. The configuration behavior of these adapters is strictly separated into a single, immutable `AdapterDescriptor` (containing Identity, Capabilities, Execution, Attention, Cache, Hardware, Verification) rather than individual capability-fetching methods.

## 5. Translation Pipeline
The Translation Pipeline is responsible for converting the generic Physical IR into backend-specific commands.

```mermaid
graph TD
    PIR[Physical IR] --> AF[Adapter Framework]
    AF --> |Maps Nodes| BOp[Backend Operation Graph]
    BOp --> BV[Backend Validation]
    BV --> EXEC[Execution Runtime (e.g. MLX)]
```

*   **Adapter Framework:** Analyzes the Physical IR nodes and edges.
*   **Backend Operation Graph:** Generates a DAG of `BackendOperation`s specific to the target platform.
*   **Backend Validation:** Ensures the operations fit within the hardware constraints defined by the `BackendDescriptor` and current `ExecutionEnvironment`.

## 6. Reference MLX Backend
The primary backend for oMLX is built on MLX, optimized for Apple Silicon (M1/M2/M3/M4). It utilizes Metal for hardware acceleration.
