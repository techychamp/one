# Parallel Execution Architecture Guide

## Overview
OMLX Execution Engine uses a `ParallelExecutionDispatcher` to execute operations concurrently based on their scheduling constraints.
The ExecutionEngine consumes an immutable `ExecutionSchedule` generated by the GraphScheduler, grouping operations into `ExecutionGroups` that represent sets of operations with identical dependency depth, meaning they can safely be executed in parallel.

## Determinism
Concurrency happens exclusively within the boundaries of an `ExecutionGroup`. Synchronization happens at the completion of a group, which guarantees deterministic execution and honors logical dependency restrictions regardless of execution timing variations. The overall model execution order will be identical to the topological sort sequence.

## Thread Safety
- Worker threads use standard standard threading primitives, safely submitting individual `op` to backend `adapter.execute(op, context)`.
- Backend adapter handles its own internal threading constraints.
- Observability phases (`observe_phase`) track metrics and record thread IDs explicitly.
- The `ExecutionContext` is immutable, avoiding race conditions during execution.

## Future Parallelism
This design enables extending the executor pool in future updates without requiring an architectural redesign. Thread pools can be swapped out, or operations can be assigned heterogeneous thread affinities (e.g. GPU vs CPU queues).
